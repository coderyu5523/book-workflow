# 시나리오 — 사내AI비서_v3

## 이야기 배경

**주인공**: 2년차 백엔드 개발자. Python과 FastAPI로 사내 시스템을 유지보수하는 팀원이다.

**상황**: 회사(커넥트HR)에서 사내 규정, 연차, 매출 등을 직원들이 매번 인사팀과 총무팀에 물어보는 비효율이 반복된다. 팀장이 "요즘 AI 좋다던데, 사내 문서 넣으면 알아서 답변해주는 비서 하나 만들어봐"라고 지시한다. 주인공은 Python은 쓸 줄 알지만 LLM이나 RAG는 이름만 들어본 수준이다.

**갈등/동기**: "ChatGPT처럼 만들면 되는 거 아닌가?"라는 막연한 기대로 시작하지만, LLM이 사내 규정에 대해 엉뚱한 답을 지어내는 것을 보고 현실을 직면한다. 팀장은 "다음 달 전사 시연"이라는 마감을 걸어놓았다. 하나씩 문제를 풀어가며 시스템을 키워가는 과정이 이 책의 줄기다.

## 시간축 + 챕터 구성

### CH01: 엉뚱한 대답 — 1일차
- 코드: ex01
- 문제: LLM에게 "신입사원 연차 규정"을 물었더니 법률 조항을 지어낸다 (환각). 우리 회사 규정이 아니라 학습 데이터에서 가져온 일반론이다.
- 해결: Context Injection(문서를 프롬프트에 직접 넣기)으로 임시 해결 --> 문서가 길어지면 토큰 한계에 부딪힘 --> RAG 개념 증명(더미 3건으로 벡터 검색 + 답변). 청킹 유무 비교로 "왜 잘라야 하는지" 체감. RAG로 해결되는 듯했으나, 추론이 필요한 복잡한 질문에서 다시 한계를 보여준다.
- 스냅샷: 스크립트 5개. LLM 단독 / Context Injection / RAG(청킹) / RAG(미청킹) / RAG+추론 질문. 아직 서버도 DB도 없는 단발성 스크립트 상태.
- 다음으로: "이걸 여러 사람이 쓰려면 서버가 필요하다."

### CH02: 기반을 닦다 — 2일차
- 코드: ex02
- 문제: 연차 잔여, 매출 조회 같은 "숫자 데이터"는 문서 검색으로 답할 수 없다. 구조화된 데이터를 다루는 시스템이 먼저 있어야 한다.
- 해결: FastAPI + PostgreSQL로 직원/연차/매출 CRUD API를 만든다. dataclass로 모델을 정의하고, Pydantic으로 입출력을 검증하고, Context Manager로 DB 연결을 관리한다.
- 스냅샷: REST API 서버 가동. `/api/employees`, `/api/leave-balance`, `/api/sales` 엔드포인트. 정형 데이터 조회 가능. 비정형(문서) 검색은 아직 없음.
- 다음으로: "API로 숫자는 꺼낼 수 있게 됐다. 그런데 사내 문서를 물어보면?"

### CH03: 문서를 모으다 — 3일차
- 코드: (개념 챕터, 코드 없음)
- 문제: "사내 문서"라고 하지만 PDF, DOCX, XLSX가 뒤섞여 있고, 파일명도 제각각이다. 아무거나 벡터DB에 넣으면 검색 품질이 떨어진다.
- 해결: 문서 종류별 수집 전략, 파일명 규칙, 메타데이터 표준, 섹션 헤더 규칙을 설계한다. "쓰레기를 넣으면 쓰레기가 나온다"는 원칙.
- 스냅샷: 문서 수집 전략 문서 완성. data/docs/ 디렉토리에 HR/Finance/Ops/Security 4개 부서 문서가 표준에 맞게 정리됨.
- 다음으로: "문서가 준비됐으니 이제 기계가 읽을 수 있는 형태로 바꿔야 한다."

### CH04: 기계가 읽는 법 — 4~5일차
- 코드: ex04
- 문제: PDF/DOCX/XLSX를 그대로 LLM에 넣을 수 없다. 텍스트를 추출하고, 적절한 크기로 자르고, 벡터로 변환해서 검색 가능하게 만들어야 한다.
- 해결: 3단계 파이프라인. (1) pypdf/python-docx/openpyxl로 텍스트 추출, (2) Fixed-size 청킹(500자, 100자 오버랩), (3) ko-sroberta-multitask 임베딩 + ChromaDB 저장. CLI 검색으로 품질 검증.
- 스냅샷: VectorDB 구축 완료. `cli_search.py`로 "연차 규정"을 검색하면 유사도 점수와 함께 관련 청크가 반환됨. 하지만 아직 대화형 인터페이스는 없음.
- 다음으로: "검색은 되는데, 이걸 자연어 답변으로 바꿔야 한다."

### CH05: 질문에 답하다 — 6~7일차
- 코드: ex05
- 문제: 벡터 검색은 관련 청크를 꺼내줄 뿐이다. 사용자가 원하는 건 "검색 결과"가 아니라 "답변"이다.
- 해결: LCEL 파이프라인(Retriever|Prompt|LLM|Parser)으로 RAG 체인을 조립한다. 출처 강제 프롬프트로 근거 없는 답변을 차단한다. WindowMemory(k=5)로 멀티턴 대화를 지원한다. FastAPI로 채팅 API를 제공한다.
- 스냅샷: RAG Q&A 엔진 완성. POST `/api/chat`으로 질문하면 답변 + 출처가 반환됨. 세션별 대화 맥락 유지. 하지만 "홍길동 연차 몇 일 남았어?"는 답변 불가 (DB 조회가 안 되므로).
- 다음으로: "문서 질문은 되는데, 숫자 데이터를 물어보면 모른다고 한다."

### CH06: 하나로 합치다 — 8~9일차
- 코드: ex06
- 문제: CH02의 DB 조회와 CH05의 문서 검색이 따로 논다. 사용자는 "홍길동 연차 몇 일이야?"와 "출장비 정산 절차가 뭐야?"를 같은 창에서 물어보고 싶다.
- 해결: 3단계 QueryRouter(규칙 --> 스키마 --> LLM)로 질문 유형을 분류한다. @tool 4개(leave_balance, sales_sum, list_employees, search_documents)를 정의한다. ReAct Agent(create_tool_calling_agent + AgentExecutor)가 라우팅 결과에 따라 도구를 선택하고 실행한다.
- 스냅샷: 통합 에이전트 가동. 정형+비정형 질문을 모두 처리. 18개 시나리오 테스트 통과. 하지만 같은 질문을 반복하면 매번 LLM을 호출하고, 에러가 나면 그대로 실패한다.
- 다음으로: "시연 전에 이걸 믿고 쓸 수 있게 만들어야 한다."

### CH07: 믿고 쓸 수 있게 — 10~11일차
- 코드: ex07
- 문제: 같은 질문에 매번 수 초씩 대기하고, LLM이 한 번 실패하면 사용자에게 에러가 그대로 노출된다. 토큰 비용도 추적이 안 된다.
- 해결: (1) ResponseCache(TTL) + EmbeddingCache(파일)로 중복 호출을 줄인다. (2) 재시도 로직(3회, 2초 간격)으로 일시적 장애를 흡수한다. (3) TokenTracker로 모델별 비용을 추적한다. (4) JSON 구조화 로깅으로 운영 가시성을 확보한다. (5) @tool을 개별 파일로 모듈화한다.
- 스냅샷: 운영 안정화 완료. 캐시 적중률, 토큰 사용량, 응답 시간을 모니터링 가능. ConnectHRAgent 클래스로 정리. 전사 시연을 앞둔 상태.
- 다음으로: "시연은 성공했다. 그런데 전사 배포가 결정되면서 실사용자 피드백이 쏟아지기 시작한다."

### CH08: 엉뚱한 문서를 가져온다 — 다음 주
- 코드: ex08
- 문제: 전사 배포 후 실사용자들이 실제 업무 질문을 던지자, 엉뚱한 문서가 검색되는 경우가 드러난다. "보안 정책을 물었는데 인사규정이 나왔다"는 피드백이 올라온다. 청크 크기가 적절한지, 검색 결과의 순서가 맞는지 확신이 없다.
- 해결: 3가지 실험. (1) 청킹 전략 실험: Fixed vs Recursive vs Semantic, 크기/오버랩 조합 비교. (2) Cross-Encoder 리랭킹: 검색 결과를 LLM이 다시 정렬. (3) 하이브리드 검색: BM25(키워드) + Vector(의미) 검색을 alpha 가중치로 결합.
- 스냅샷: 3가지 튜닝 실험 완료. 각 실험의 검색 품질 개선을 수치로 확인. 최적 파라미터 조합을 선정. 하지만 "기본 Retriever로는 한계가 있는 질문"이 남아 있다.
- 다음으로: "검색기 자체를 바꿔볼 수는 없을까?"

### CH09: 검색의 한계를 넘다 — 그 다음 주
- 코드: ex09
- 문제: 긴 문서에서 맥락을 잃거나, 메타데이터 필터가 필요하거나, "WFH"같은 약어를 이해 못 하는 등 기본 Retriever로는 풀 수 없는 문제가 있다.
- 해결: 2가지 실험 그룹. (1) 고급 Retriever 3종: ParentDocumentRetriever(큰 문서 복원), SelfQueryRetriever(메타데이터 필터), ContextualCompressionRetriever(답변 압축). (2) Query Rewrite 3종: 약어/동의어 확장, HyDE(가상 답변으로 검색), Multi-Query(질문 다각화).
- 스냅샷: 고급 검색 기법 실험 완료. 각 기법이 "어떤 유형의 질문"에 효과적인지 정리. 하지만 "애초에 문서 파싱이 잘못됐으면?"이라는 근본적 질문이 남는다.
- 다음으로: "검색 품질의 천장은 결국 파싱 품질이다."

### CH10: 끝까지 의심하다 — 마지막 주
- 코드: ex10
- 문제: 이미지 기반 PDF에서 텍스트가 누락되고, 표가 깨지고, 파싱 품질을 객관적으로 측정할 방법이 없다.
- 해결: 3가지 실험. (1) 문서 파싱 비교: OCR(EasyOCR) vs Vision LLM(LLaVA/Qwen2-VL). (2) 하이브리드 파싱: OCR 결과가 짧으면 Vision LLM으로 보완하는 전략. (3) 평가 프레임워크: Precision@K, Recall@K, MRR, Hallucination Rate로 RAG 전체 품질을 정량 측정.
- 스냅샷: 전체 RAG 파이프라인 완성. 파싱부터 검색, 답변, 평가까지 순환 개선 체계 구축. "ConnectHR"이 실무에 배포 가능한 수준.
- 다음으로: (에필로그) "이제 진짜 시작이다."

## 학습 순서 근거

1. **CH01이 첫 번째인 이유**: 환각 체험이 이후 모든 기술 도입의 동기다. "왜 RAG가 필요한지"를 체감하지 않으면 이후 챕터가 정답부터 알려주는 교과서가 된다.

2. **CH02가 CH04보다 앞서는 이유**: CH06에서 정형+비정형을 통합할 때 DB가 이미 있어야 한다. CRUD는 대부분의 개발자에게 익숙한 영역이라 난이도 곡선을 완만하게 유지한다.

3. **CH03(개념)이 CH04 앞에 오는 이유**: "쓰레기를 넣으면 쓰레기가 나온다"를 먼저 인식해야 파싱/청킹의 의미가 산다. 코드 없이 전략만 다루는 쉼터 역할도 한다.

4. **CH04 --> CH05 순서의 이유**: 벡터DB가 있어야 RAG 체인을 조립할 수 있다. CH04의 store.py가 CH05의 build_retriever()가 참조하는 ChromaDB를 만든다.

5. **CH05 --> CH06 순서의 이유**: RAG 단독 동작을 먼저 확인한 뒤에야 "DB 조회와 합치자"는 욕구가 자연스럽다. Agent/Tool Calling은 LangChain의 고급 개념이므로 LCEL을 먼저 익혀야 한다.

6. **CH06 --> CH07 순서의 이유**: 통합 에이전트가 "동작은 하지만 불안정"한 상태를 경험해야 캐시, 재시도, 모니터링의 필요성이 와닿는다.

7. **CH08~10이 마지막 블록인 이유**: CH07까지가 "만드는" 과정이라면, CH08~10은 "개선하는" 과정이다. 완성된 시스템이 있어야 "어디가 약한지" 실험할 수 있다. 순서는 검색 파라미터(CH08) --> 검색 구조(CH09) --> 입력 품질+평가(CH10)로, 튜닝 포인트가 점점 깊어진다.

8. **CH08~10 사이의 순서**: (1) CH08(청킹/리랭킹/하이브리드)은 기존 Retriever의 파라미터를 조정하는 수준이라 진입 장벽이 낮다. (2) CH09(고급Retriever/QueryRewrite)는 Retriever 자체를 교체하거나 질문을 변환하는 구조적 변경이라 한 단계 더 깊다. (3) CH10(파싱+평가)은 "입력 데이터 품질"이라는 가장 근본적인 층위를 다루며, 평가 프레임워크로 전체를 정량적으로 되돌아본다.

## 비유 소재

**버전업형 비유: "가게 운영"**

ConnectHR 시스템의 성장을 "가게를 열고 키우는 과정"에 비유한다.

| 챕터 | 비유 확장 |
|------|----------|
| CH01 | 길에서 행인에게 물어보기 (아무나 붙잡고 질문 = LLM 단독). 메뉴판을 보여주며 물어보기 (Context Injection). 전문 안내원 고용 (RAG). |
| CH02 | 가게 내부 시스템 구축. 재고 장부(DB), 주문 접수 창구(API)를 만든다. |
| CH03 | 진열할 상품(문서)을 정리한다. 라벨이 안 붙은 상품은 찾을 수 없다. |
| CH04 | 상품을 선반에 진열한다. 크기별로 나누고(청킹), 바코드를 붙이고(임베딩), 선반에 넣는다(ChromaDB). |
| CH05 | 안내 데스크를 연다. 손님이 물어보면 선반에서 찾아와서(Retriever) 설명해준다(LLM). |
| CH06 | 안내 데스크와 재고 창구를 하나로 합친다. 손님 질문 유형에 따라 담당자를 배정한다(Router + Agent). |
| CH07 | 가게를 안정화한다. 자주 묻는 질문 답변을 게시판에 붙이고(캐시), 매출을 기록하고(모니터링), 직원 교대 시 인수인계를 한다(로깅). |
| CH08 | 선반 배치를 최적화한다. 상품 크기를 바꿔보고(청킹), 추천 순서를 조정하고(리랭킹), 검색 방식을 바꿔본다(하이브리드). |
| CH09 | 가게의 한계를 넘는다. 작은 샘플로 찾은 뒤 원래 상품 전체를 꺼내오고(ParentDoc), 태그 검색을 추가하고(SelfQuery), "그거 있잖아 그거"라는 모호한 요청도 이해한다(QueryRewrite). |
| CH10 | 품질 관리를 도입한다. 상품 라벨이 정확한지 확인하고(파싱 비교), 자동 품질 검사 라인을 만든다(평가 프레임워크). |
